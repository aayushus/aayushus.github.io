Setup Guide
Follow this guide in order, top to bottom. Every step is required unless marked optional. By the end, all six agents will be running and your SDLC will be AI-first.
⏱ Total time: ~2.5 hrs (admin) + 30 min per repo
👤 Who does this: Engineering Lead (admin steps) + each engineer (Section 10)
📋 You need: Devin Teams/Enterprise, GitHub org admin, Jira admin, Slack admin
01
Prerequisites
Confirm you have everything before starting. Don't skip this — you'll hit blockers mid-setup otherwise.
⏱ 5 min
ℹ️
Check each item off as you confirm it. Do not proceed until all are checked.
02
Connect Integrations
Wire Devin to GitHub, Jira, and Slack. Do all three before moving on.
⏱ 20 min

2A — GitHub

1
Open app.devin.ai → Settings → Integrations → GitHub
2
Click Connect
You'll be redirected to GitHub to install the Devin GitHub App.
3
Select your organization (not "Only select repositories")
Grant access to all repositories. Devin needs read/write access to open PRs, push branches, and read repo contents.
4
Confirm the green Connected status in Devin settings

2B — Jira

1
Open app.devin.ai → Settings → Integrations → Jira
2
Click Connect and authorize with your Jira admin account
3
Create a Devin service account in Jira
In Jira admin: User Management → Create user. Suggested email: devin@yourcompany.com. This account is what Devin uses to post comments and update tickets — so they appear as "Devin" not as you.
4
In Jira admin: create an OAuth 2.0 app for the service account
Atlassian admin → Security → OAuth 2.0 integration → Create. Scopes needed: read:me, read:jira-user, read:jira-work, write:jira-work
5
Back in Devin: click Connect service account and enter the client ID + secret
6
Create these labels in your Jira project
Go to your Jira project → Project settings → Labels → Create: devin, !implement, !write-tests, !design-review, !security-review

2C — Slack

1
Open app.devin.ai → Settings → Integrations → Slack
2
Click Connect → Add to Slack and authorize for your workspace
3
Invite @Devin to all seven channels
In each channel, type: /invite @Devin
Channels: #dev-runs #code-review #qa-alerts #releases #engineering #security #incidents
4
Test the integration
In #engineering, type: @Devin hello — you should get a response within 30 seconds.
03
Enable Devin Review
Turns on automated PR review across all repos. This is what powers Zeus.
⏱ 5 min
1
Open app.devin.ai → Settings → Review
2
Under Repositories, click the dropdown and add all your repos
Add every repo the team works in. Devin will auto-review all PRs on these repos.
3
Toggle Auto-review all PRs to ON
This triggers a review every time a PR is opened, new commits are pushed, or a draft is marked ready. If you want to start conservative, use "Auto-review Devin-authored PRs only" first and expand later.
4
Enable Insert link in PR description (default is ON, leave it)
This adds a Devin Review link to every PR description so reviewers can jump straight to the diff view.
💡
Fast review shortcut: For any GitHub PR URL, replace github.com with devinreview.com to open the full Devin Review view instantly. Share this with your team.
04
Index Repositories
Indexing enables Ask Devin, DeepWiki, and codebase-aware context in every agent session.
⏱ 10 min (indexing runs in background)
1
Open app.devin.ai → Settings → Repositories
2
Click Index next to each repo
Indexing runs in the background and takes 1–10 minutes per repo depending on size. You'll see a green status when complete.
3
Verify: open deepwiki.com and search for your repo
If indexed, you'll see auto-generated architecture docs for your repo. If not yet visible, wait a few minutes and refresh.

You can also trigger indexing via API: PUT /api/v3/organizations/{orgId}/repositories/bulk-index if you have many repos to index at once.

Hermes — Development Agent
Implements Jira tickets autonomously
!implement
NameImplement
Macro!implement
WhereSettings → Library → Playbooks → Create
Instructions — copy and paste into Devin
You are implementing a Jira ticket. Follow these steps exactly. 1. READ THE TICKET Read the Jira ticket description, acceptance criteria, and all linked design documents carefully. If acceptance criteria is missing or unclear, post a comment on the Jira ticket asking for clarification and STOP — do not proceed until clarified. 2. UNDERSTAND THE CODEBASE Check AGENTS.md in the repository root for architecture conventions, naming patterns, and constraints. Search the codebase for similar existing implementations before writing new code — follow established patterns. 3. CHECK SKILLS List available skills with @skills. You will use @skills:test-before-pr before opening the PR. Do not skip this. 4. IMPLEMENT Cut a feature branch: feature/[TICKET-ID]-short-kebab-description Implement the feature end-to-end: - Follow existing patterns — do not invent new conventions - Write unit tests and integration tests as you implement, not after - Never leave console.log, commented-out code, or TODO comments in the PR - If you discover the ticket is underspecified mid-implementation, post a Jira comment asking for clarification before proceeding 5. TEST Run @skills:test-before-pr. Fix ALL failures before continuing. Do not open a PR with failing tests. 6. OPEN PR - Title: [TICKET-ID] Short description of what changed - Description: what changed, why, and how to test it manually - Link the Jira ticket URL in the PR description - Assign 2 reviewers 7. UPDATE JIRA Post the PR link as a comment on the Jira ticket. Move the ticket status to "In Review". 8. NOTIFY Post the PR link to the #dev-runs Slack channel with one sentence summarising what was implemented.

Zeus — Code Review Agent
Reviews all open PRs and triages CI failures
!review
NameReview
Macro!review
WhereSettings → Library → Playbooks → Create
Instructions — copy and paste into Devin
You are performing code review on all open pull requests in the organisation. FOR EACH OPEN PR: 1. Read the linked Jira ticket and its acceptance criteria. Understand what the PR is supposed to do before looking at the diff. 2. Read REVIEW.md in the repository root. Apply every rule defined there. 3. Review the diff for: 🚨 SEVERE bugs — high-confidence issues that will cause failures in production ⚠️ NON-SEVERE bugs — potential issues worth fixing before merge 🔒 SECURITY — injection, hardcoded secrets, insecure token handling, missing input validation, error disclosure 📊 COVERAGE — flag if test coverage has dropped 💥 BREAKING CHANGES — API changes that affect downstream consumers 4. Post findings as inline GitHub review comments using this format: 🚨 Severe: [what the issue is and why it will fail] ⚠️ Non-severe: [what should be fixed] 💡 Suggestion: [optional improvement] 5. If CI is failing: investigate the root cause. Post a review comment explaining what is failing and why. 6. Auto-approve if: the PR only changes documentation, config files, or dependency patch versions with no API changes. Post a note explaining why it was auto-approved. AFTER REVIEWING ALL PRS: Post a summary to #code-review: - List each PR: author, title, status (needs attention / approved / blocked) - Flag any PRs open more than 48 hours - Flag any PRs with Severe findings that must be fixed before merge

Argus — QA & Test Agent
Runs nightly regression and files Jira bugs
!test
NameTest
Macro!test
WhereSettings → Library → Playbooks → Create
Instructions — copy and paste into Devin
You are running the QA and regression suite. Follow these steps. 1. PULL LATEST Check out the latest main branch. Confirm the branch is up to date. 2. RUN TESTS Run the full test suite with coverage: npm install && npm test -- --coverage (Adapt the command to the actual package manager and test runner in use.) 3. ANALYSE RESULTS For each FAILED test, determine: - Real regression: test was passing before a recent commit that broke it - Flaky test: passes sometimes, fails sometimes, not caused by a code change 4. FILE JIRA TICKETS FOR REAL REGRESSIONS For each real regression, create a Jira bug ticket: - Title: [Regression] [test name] failing since [commit hash] - Description: what the test verifies, why it is failing, the full stack trace - Steps to reproduce the failure - Priority: P1 if it affects a critical user-facing path, P2 otherwise - Label: regression For flaky tests: post a note to #qa-alerts only — do NOT file a Jira ticket. 5. CHECK COVERAGE If any module's coverage has dropped below 80%, flag it in the report. If any module is below 60%, file a Jira ticket: - Title: [Coverage] [module name] below threshold - Priority: P2 6. POST SUMMARY TO #qa-alerts ✅ X tests passed ❌ Y failed (Z real regressions, W flaky) 📊 Overall coverage: X% 🔴 Modules below 80%: [list or "none"] 📎 New Jira tickets: [links or "none"]

Atlas — Release Agent
Weekly changelog, GitHub tag, Confluence, Jira close
!release
NameRelease
Macro!release
WhereSettings → Library → Playbooks → Create
Instructions — copy and paste into Devin
You are performing the weekly release process. Follow every step in order. Do NOT skip any step. Do NOT proceed if a step fails — post to #releases and stop. STEP 1 — CHECK BLOCKERS Search Jira for open tickets with priority P1 or P2 that are unresolved. If any exist: post to #releases: "⚠️ Release blocked: [ticket list]" and STOP. STEP 2 — VERIFY CI Confirm CI is green on the main branch. If failing: post to #releases: "⚠️ Release blocked: CI failing on main" and STOP. STEP 3 — COMPILE CHANGELOG Find all pull requests merged to main since the last release tag. For each PR, retrieve the linked Jira ticket description if available. Categorise into: ✨ Features — new functionality 🐛 Bug Fixes — issues resolved ⚡ Performance — speed or efficiency improvements 🔧 Maintenance — dependencies, refactors, infrastructure STEP 4 — CREATE GITHUB RELEASE Tag the release: vYYYY.MM.DD (today's date, e.g. v2026.06.13) Create a GitHub Release with the categorised changelog as the body. STEP 5 — UPDATE CONFLUENCE Find the "Release Notes" page in Confluence. Add a new section at the top for this release with the full changelog. STEP 6 — CLOSE JIRA VERSION Find the current sprint release version in Jira. Mark all resolved tickets as belonging to this version. Close the version. STEP 7 — NOTIFY Post to #releases with the full changelog, the GitHub release link, and a one-sentence summary of the most significant change this release.

Hephaestus — Maintenance Agent
Dependencies, dead code, security audit, lint
!maintenance
NameMaintenance
Macro!maintenance
WhereSettings → Library → Playbooks → Create
Instructions — copy and paste into Devin
You are performing the weekly maintenance sweep across all repos. 1. DEPENDENCY UPDATES Run: npm outdated (adapt for the package manager in use) Patch updates (1.2.3 → 1.2.4): open a PR immediately. Title: chore: update [packages] patch versions Group multiple patches into one PR per repo. Minor updates (1.2.3 → 1.3.0): open a PR with a review note. Title: chore: update [package] to [version] (minor) Add note: "Minor update — review changelog for any behaviour changes." Major updates (1.2.3 → 2.0.0): open a PR, assign to a senior engineer. Title: chore: update [package] to [version] (MAJOR — needs review) Include a summary of breaking changes from the changelog. ⚠️ Do NOT auto-merge major updates. NEVER auto-merge updates to packages touching auth, payments, or database connections — always flag these for human review. 2. SECURITY AUDIT Run: npm audit (or equivalent) For each HIGH or CRITICAL vulnerability, file a Jira ticket: - Title: [Security] CVE-XXXX in [package] — [severity] - Priority: P1 for CRITICAL, P2 for HIGH - Include: CVE ID, affected version, fix version, one-line impact assessment 3. DEAD CODE Look for: exported functions with no imports, files with no references, feature flags that are always-true or always-false. Open a PR to remove clearly unused code — one PR per module. 4. LINT Run: npm run lint --fix Commit auto-fixed issues: chore: auto-fix lint issues 5. REPORT TO #engineering 📦 Dependencies: X patch PRs, Y minor PRs, Z major PRs (need review) 🔒 Security: X CVEs filed (links) 🧹 Dead code: X removal PRs opened ✨ Lint: X issues auto-fixed

Themis — Security & Compliance Agent
Security reviews on sensitive PRs + weekly sweep
!security-review
NameSecurity Review
Macro!security-review
WhereSettings → Library → Playbooks → Create
Instructions — copy and paste into Devin
You are performing a security review. This applies to PRs touching sensitive paths AND as a weekly codebase sweep. FOR PR SECURITY REVIEW: 1. Read REVIEW.md for this repository's specific security rules. 2. Check for each of the following. Flag every finding with severity: CRITICAL — exploitable now, must fix before merge HIGH — significant risk, must fix before merge MEDIUM — should be fixed, can merge with a tracked ticket LOW — informational, suggest fix Checks: - Injection: SQL, command, LDAP injection via string concatenation - Hardcoded secrets: API keys, passwords, tokens in source code - Insecure direct object references: user-controlled IDs accessing records without authorisation checks - Missing input validation: public endpoints accepting raw user input - Insecure token handling: tokens in URLs, logged to console, stored in localStorage, or short expiry not enforced - Error disclosure: stack traces or internal details in API responses - Broken authentication: JWT validation bypasses, session fixation, missing logout invalidation 3. Post findings as GitHub review comments. Format: 🔴 CRITICAL: [file:line] [what it is] [how it can be exploited] [fix] 🟠 HIGH: [same format] 🟡 MEDIUM: [same format] 4. File a Jira security ticket for MEDIUM and above: Title: [Security] [vulnerability type] in [file/component] Priority: P1 for CRITICAL/HIGH, P2 for MEDIUM FOR WEEKLY SWEEP (no PR): 1. Scan all repos for hardcoded secrets (API key patterns, token strings) 2. Check for packages with known CVEs: npm audit 3. Review recent commits (last 7 days) touching auth or authorisation code 4. Post summary to #security with all findings and Jira ticket links

08
Per-Repo Files
Add these four files to every repo. Devin reads them automatically at the start of every session. Commit and push — no Devin config needed.
⏱ 30 min per repo
⚠️
The content below is a template. Customise it to match your actual tech stack, architecture, and conventions before committing. Devin takes these files literally — generic instructions produce generic output.

AGENTS.md

Place in repo root. Devin reads this at the start of every session in this repo. Document your architecture, naming conventions, and constraints.

AGENTS.md
# Team AGENTS.md — Read this before starting any task in this repo ## Architecture Conventions - Services communicate via REST externally, gRPC internally - No direct database access from the API layer — use the repository pattern - All new services require an OpenAPI spec committed to /docs before coding starts - Database migrations must be backward-compatible (never drop columns, use defaults) ## Tech Stack - Backend: Node.js 20 + TypeScript 5 - Database: PostgreSQL 15 via Prisma ORM - Frontend: React 18 + Vite - Testing: Jest (unit), Playwright (e2e) - CI: GitHub Actions ## Naming Conventions - Files: kebab-case (user-service.ts) - Classes: PascalCase - Functions and variables: camelCase - Database tables: snake_case - Environment variables: SCREAMING_SNAKE_CASE ## Things You Must Never Do - Never commit secrets, API keys, or credentials to the repo - Never use `any` type in TypeScript — use proper types or `unknown` - Never write raw SQL — always use the Prisma ORM - Never merge to main directly — always use a PR - Never skip tests to make CI pass faster — fix the test ## Branch Naming feature/[JIRA-KEY]-short-description bugfix/[JIRA-KEY]-short-description chore/short-description ## Team Playbook Full SDLC operating procedure: https://aayushus.github.io/sdlc/

REVIEW.md

Place in repo root. Zeus reads this on every PR review. Define what to always look for, what to always flag, and what to ignore.

REVIEW.md
# Code Review Guidelines ## Critical Paths — Always Review These Carefully - `src/auth/` — any change must be reviewed for auth bypass, token handling, session issues - `src/payments/` — review for injection, PCI compliance, idempotency - `src/database/migrations/` — must be backward-compatible; check for destructive changes - `src/api/routes/` — check input validation on every public endpoint ## Security Checks — Flag Every Instance - Raw SQL string concatenation anywhere in the codebase - Any secret, token, or password-like string hardcoded in source files - User-controlled IDs used in DB queries without authorisation check - Error responses that include stack traces or internal system details - `console.log` statements containing user data or tokens ## Code Quality Checks - TypeScript `any` type usage — flag every instance, suggest proper type - Functions longer than 50 lines — suggest refactoring - Deeply nested callbacks (>3 levels) — suggest async/await or extraction - Missing error handling on async operations - Test coverage dropping below 80% on changed files ## Anti-Patterns — Flag These - Direct database access from API route handlers (bypass repository layer) - Business logic inside React components (should be in hooks or services) - Synchronous file I/O in request path - N+1 database queries in loops ## What to Ignore - `src/generated/` — auto-generated files, do not review - `*.lock` files — only flag if the diff looks unusual - `.github/workflows/` — only flag security issues (secret exposure, etc.) - Whitespace-only changes

.agents/skills/test-before-pr/SKILL.md

Create the directory path and file. Devin invokes this automatically before opening any PR. Adapt the commands to your actual test runner.

.agents/skills/test-before-pr/SKILL.md
--- name: test-before-pr description: Run all quality checks before opening a pull request. Always invoke this before creating a PR. --- ## Pre-PR Checklist Run these steps in order. Stop and fix any failures before continuing. Do NOT open a PR if any step fails. ### 1. Install dependencies ``` npm install ``` ### 2. Lint — fix all issues ``` npm run lint ``` If lint fails, run `npm run lint --fix` to auto-fix, then commit the fixes. If issues remain after auto-fix, resolve them manually before continuing. ### 3. Type check ``` npx tsc --noEmit ``` Fix all TypeScript errors. Do not use `@ts-ignore` to bypass them. ### 4. Run tests ``` npm test ``` All tests must pass. If a test was already failing before your changes, note it in the PR description but do not leave your changes untested. ### 5. Build ``` npm run build ``` The build must succeed with no errors. ### Done All steps passed. You may open the PR.

09
Knowledge Base
Knowledge items are org-wide context that Devin recalls automatically. Go to Settings → Knowledge → Add Knowledge for each entry.
⏱ 15 min

For each entry: enter the Trigger Description (what Devin is working on when it should recall this) and the Content. Pin each one to "all repos".

1
Trigger: when designing or implementing anything involving the database
Database conventions: - Always use Prisma ORM. Never write raw SQL. - All migrations must be backward-compatible. Never drop a column directly — deprecate it first, remove it in a later release. - Use database transactions for any operation that touches more than one table. - Index foreign keys and any column used in a WHERE clause on large tables. - Never run migrations in the application startup path.
2
Trigger: when working on authentication, authorisation, or security
Security conventions: - Never hardcode secrets in source code. Use environment variables only. - JWTs expire in 15 minutes. Refresh tokens expire in 7 days. - All user inputs must be validated before reaching the database or file system. - Use parameterised queries everywhere. Never concatenate user input into a query string. - Passwords are hashed with bcrypt, cost factor 12. - Failed login attempts are rate-limited at 5 per 5 minutes per IP. - Auth changes require Themis security review before merge.
3
Trigger: when writing tests or checking test coverage
Testing conventions: - Minimum coverage threshold: 80% overall, 90% for auth and payments modules. - Unit tests go in __tests__/ directories alongside the source file. - Integration tests go in tests/integration/. - E2E tests go in tests/e2e/ and use Playwright. - Test file naming: [filename].test.ts for unit, [feature].spec.ts for e2e. - Never mock the database in integration tests — use a test database instance. - Use factories (from tests/factories/) to create test data, not raw DB inserts.
4
Trigger: when implementing a new API endpoint or modifying an existing one
API conventions: - All public endpoints must be documented in the OpenAPI spec at /docs/openapi.yaml before or alongside implementation. Not after. - API versioning: all endpoints are prefixed /api/v1/. Breaking changes require a new version prefix. - All endpoints must validate input using Zod schemas. - Error responses follow this format: { "error": { "code": "ERROR_CODE", "message": "Human readable message" } } - Never expose internal IDs in public APIs — use UUIDs. - Rate limit all public endpoints: 100 req/min per user by default.
